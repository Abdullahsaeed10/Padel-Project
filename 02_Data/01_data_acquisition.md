## 2.1 Data Acquisition — where the data comes from

The app combines two data layers, each with its own source.

### Layer A — Pro Tour data (real)

**Primary source:** the [Padel API](https://padelapi.org) (Fantasy Padel Tour). It exposes
Premier Padel rankings and match results via a REST endpoint with token-based auth.
The free tier covers 50,000 requests/month (10 req/min) and the last 6 months of matches —
more than enough for a student project. The app fetches this data live at runtime
(`05_App/utils.py`: `load_rankings` / `load_matches`, men's and women's tours). The
`api_client.py` file in this folder is the standalone command-line equivalent —
`fetch_rankings(category)` and `fetch_matches(category)` return JSON which is flattened to
the same CSV schema and written to `05_App/data/` to pre-warm the offline snapshots.

**Secondary / backup sources** (used if the API is unreachable, e.g. during the live demo):
- The [Kaggle "Padel Tennis World Championship" dataset](https://www.kaggle.com/datasets/mathurinache/padel-tennis-world-championship) — smaller but real.
- The Premier Padel public results page (scrapable with `requests + BeautifulSoup`).
- A committed `pro_matches.csv` generated by `generate_padel_data.py`, modeled on real
  2024-2025 tournaments (Riyadh P1, Qatar Major, Rome Major, …) with real player names and
  partnerships, plausible scores. This is what runs in the offline demo.

> **What to say in the presentation:** *"For the pro data I built on top of the Padel API,
> which gives me Premier Padel rankings and match results with a free tier. To make the
> live demo robust against wifi issues, I cache the data locally as CSV — the committed
> CSV is structured exactly like the API response, so swapping in real fetched data is a
> one-line change."*

### Layer B — Personal Match Log (synthetic seed)

There is **no public dataset of amateur padel match logs** — and that absence is precisely
what motivates the personal-journal half of the project (see brief §1.1). The personal CSV
is **seed data** that demonstrates what real users will produce once they start logging.

Following the lecture's advice on data issues ("Embrace Data Issues — if the data isn't
there, make the absence part of the story"), this is acknowledged transparently in the
presentation rather than hidden.

The synthetic log was generated with deliberate, documented patterns the app can later
surface (see §2.3 EDA below).

## 2.2 Examination & Transformation

### Files produced

```
02_Data/
├── generate_padel_data.py    # reproducible generator
├── api_client.py              # production fetcher (Padel API)
└── data/
    ├── pro_players.csv         30 rows · top male Premier Padel players
    ├── pro_matches.csv        276 rows · 12 tournaments × 23 matches
```

### Schema — `pro_players.csv`

| column | type | notes |
|---|---|---|
| player_id | int | 1..30, current ranking position |
| name | string | real player |
| country | string | 3-letter code |
| side | enum | D = drive (right court), R = reves (left court) |
| hand | enum | left / right |
| height_cm | int | |
| birth_year | int | |
| ranking_points | int | Premier Padel ranking points |

### Schema — `pro_matches.csv`

| column | type | notes |
|---|---|---|
| match_id | int | |
| date | ISO date | |
| tournament | string | e.g., "Rome Major" |
| category | enum | P1 / P2 / Major / Finals |
| surface | enum | indoor / outdoor |
| country | string | 3-letter code |
| round | enum | R32 / R16 / QF / SF / F |
| team{1,2}\_p{1,2} | string | player names |
| winner_team | enum | 1 / 2 |
| set{1,2,3} | string | "6-3" format, from winner's perspective; set3 blank if straight sets |
| duration_min | int | |

### Schema — `my_matches.csv`

| column | type | notes |
|---|---|---|
| match_id | int | |
| date | ISO date | |
| partner | string | one of 6 partners |
| opponents | string | "Name & Name" |
| club | string | venue |
| surface | enum | indoor / outdoor |
| sets_played | int | 2 or 3 |
| set{1,2,3}\_us / \_them | int | games won by each side, set3 blank if straight |
| result | enum | W / L |
| winners\_{forehand, backhand, smash, volley, bandeja} | int | shot tally per category |
| errors\_{forehand, backhand, smash, volley, bandeja} | int | |
| duration_min | int | |
| notes | string | free-text |

### Transformations applied

1. **Score parsing.** Set strings like `"6-3"` are split into `(my_games, opp_games)` tuples
   at load time inside the Streamlit cache so charts can work with numeric columns.
2. **Derived columns.**
   - `won = (winner_team == 1) when player on team 1 else (winner_team == 2)` for pro views.
   - `net_balance_per_shot = winners − errors` for the personal log.
   - `is_three_setter = (sets_played == 3)` for fatigue analysis.
3. **Joins.** Pro players ↔ pro matches via name (both sides of every match) for the
   player-detail page.
4. **Aggregations.** Rolling 5-match win rate, partner aggregation, surface aggregation.
   All done in pandas with `groupby` + `rolling`.

> **What to say in the presentation:** *"The transformation step is mostly about taking
> tournament-style score strings and turning them into the rectangular tidy data my charts
> want. I do all the heavy lifting once at load time inside a Streamlit cache so the app
> stays snappy when the user clicks around."*

## 2.3 Exploration (EDA) — what the data told me

A short exploratory pass over the data, written up here so the prof can see it drove the
design choices on the next page. Numbers below are generated by the seed data so they're
the ones you'll see in the live app.

*(Numbers below are the actual values produced by the generator at seed=42 — the same
ones the live app shows.)*

### Findings — Pro Tour (276 matches across 12 tournaments)

- **Straight-set rate: 68%.** Matches the real Premier Padel rate of ~68-72%. *Design
  consequence:* don't lean on third-set analytics for the pro side — most matches don't
  have one.
- **Surface split:** 161 outdoor / 115 indoor matches in the season. Average duration is
  nearly identical (~91 min). *Design consequence:* surface is a useful filter but not a
  big descriptive variable on its own.
- **Partnership stability:** 15 stable pairs cover the vast majority of matches. *Design
  consequence:* a "head-to-head between pairs" view is more meaningful than a per-player
  view.

### Findings — Personal Log (40 matches across 4 months)

- **Partner effect is huge.** Win rate with Luca = **80%** (8/10), with Davide = **20%**
  (1/5). The other four partners cluster around 50%. *This is the single most surprising
  and actionable fact in the data — it's the chart we put above the fold.*
- **Third-set fatigue is real.** Two-set match win rate = **80%**; three-set match win
  rate = **40%**. A drop of 40 percentage points when matches go long. *Design
  consequence:* a dedicated "set-by-set" view is justified.
- **Shot balance is uneven (net winners − errors over 40 matches):**
  - Bandeja: **strongest** (≈ +160 net winners)
  - Volley: strong (≈ +80)
  - Forehand: positive (≈ +65)
  - Backhand: net-negative (≈ −120)
  - Smash: net-negative (≈ −155) — the clearest weakness
  *Design consequence:* a diverging horizontal bar chart ("shot DNA") makes this readable
  at a glance.

### Why this matters for design

Each of those findings becomes one chart in the final app. The EDA is what kept the
chart count *down*: every chart has to earn its place by answering one finding.

> **What to say in the presentation:** *"I did a short EDA pass before opening Figma —
> not the other way around. Every chart you'll see later corresponds to one specific
> finding I documented at this step. That kept me from drawing pretty charts that don't
> say anything."*
